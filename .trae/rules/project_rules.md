# 日语单词注音（振假名）数据修复规则 V2.3

本文档概述了用于识别和修复N1-N5词汇数据集中日语汉字注音的系统化规则。旨在将注音尽可能标准化为**逐字注音（Monoruby）**格式，同时兼顾工程可行性与教学准确性。

---

## 0. 绝对禁令 (Absolute Prohibition)
> ⚠️ **严禁使用任何脚本（Python、Shell、Node.js 等）对 JSON 文件进行批量写入或自动替换。**  
> 所有对 `examples` 字段的修改**必须由 AI 在对话中手动逐条完成**，并立即展示前后对比，严禁静默落盘。  
> 违反此禁令将导致数据完整性风险，一经发现立即回滚。

---

## 0. 总原则 (Guiding Principle)
> **"当逐字注音在语言学或教学上存在明显争议时，优先保证正确读音，其次才是逐字一致性。"**
> 如果无法确定正确的逐字拆分，应回退到整体注音，避免错误的分割误导学习者。

### 0.1 数据完整性要求 (Data Integrity Requirements)
> **每个日语单词必须包含至少三条例句**，以确保学习者能够充分理解词汇在不同语境下的用法和含义。
> * 例句应覆盖不同的语法结构和语境
> * 例句应体现词汇的常见用法和搭配
> * 所有例句都必须遵循本规则的注音标准

## 1. 核心注音规范

### 目标格式
*   **语法结构**：`[汉字][读音]`
*   **约束**：汉字与左括号 `[` 之间不能有空格。
*   **粒度**：**严格的逐字注音**。每个汉字字符都必须跟随其专属的读音括号，除非属于"特殊情况"。

### 正确 vs 错误示例
| 类型 | 错误 (Bad) | 正确 (Good) | 理由 |
| :--- | :--- | :--- | :--- |
| **复合词** | `銀行[ぎんこう]` | `銀[ぎん]行[こう]` | 基础规则：精确对应。 |
| **动词** | `歩く[あるく]` | `歩[ある]く` | 送假名必须在括号外。 |
| **符号** | `人々[ひとびと]` | `人[ひと]々[びと]` | 叠字需注音。 |
| **外来语修饰** | `超ハイレベル` | `超[ちょう]ハイレベル` | 汉字必须注音，片假名不注音。 |
| **片假名注音** | `ビール[びーる]` | `ビール` | 片假名作为表音文字，严禁注音。 |

## 2. 上下文判断规则 (何时注音)

### 规则 A: 必须注音 (Mandatory)
1.  **汉字字符**：所有Unicode汉字区域字符。
2.  **叠字符号**：`々` 必须标注其上下文相关的读音 (如：`国々` -> `国[くに]々[ぐに]`)。
3.  **修饰性汉字**：当汉字修饰外来语时，汉字需注音 (如：`元[もと]サッカー選手`)。
4.  **作为量词后缀的汉字**：数字后的汉字若承担读音，必须注音 (如：`3人[にん]`)。

### 规则 B: 禁止/省略注音 (Prohibited)
1.  **假名**：平假名本身不注音。
2.  **片假名**：严禁为片假名（Katakana）添加平假名注音。片假名自身即为表音符号。
3.  **标点与符号**：以下符号**不参与**注音与拆分：
    *   长音符 `ー`
    *   中黑点 `・`
    *   波浪线 `〜`
    *   标点 `。`、`、` 等
4.  **数字本身**：阿拉伯数字 (`1`, `2024`) **默认不注音**。
    *   *特例*：仅当数字是熟语整体的一部分且读音特殊时（如 `1日[ついたち]`），可按熟字训整体处理，否则一律仅给具体的汉字量词注音。

## 3. 自动修复与拆分策略 (核心逻辑)

### 策略 A: 基于词典的拆分 (标准复合词)
*   **逻辑**：根据词典中单字的音读/训读库，将假名序列拆分匹配到汉字。
*   **示例**：`図書館[としょかん]` -> `図[と]書[しょ]館[かん]`

### 策略 B: 词干与送假名提取 (动词/形容词)
**关键补充：部分词干为假名的情况**
即使汉字前亦有假名，只要该汉字属于词干的一部分，就必须注音。
*   **示例**：`受け取る`
    *   `受` 是词干，`け` 是送假名，`取` 又是词干。
    *   **处理**：识别出 `け` 和 `る` 为送假名（在括号外）。
    *   **结果**：`受[う]け取[と]る`

### 策略 C: 多音字判定优先级 (Priority Logic)
当汉字存在多种读音时（如"生"、"日"），按以下优先级确定读音：
1.  **词级固定读音 (Word-Level Lock)**：如果整词在词典中有明确条目（如 `海女[あま]`），优先于单字拆分。
2.  **词性匹配**：动词/形容词优先匹配训读；名词/熟语优先匹配音读。
3.  **共现频率匹配**：检查前后汉字组合的常见度。
4.  **自然拍数拆分**：选择假名长度最均匀、最符合音律的拆分。
5.  **无法判断** -> **安全回退 (Fallback)**：保留整体注音。

### 策略 D: 混合内容边界定义 (Mixed Content)
当一个词本身是"汉字+假名"的混合体时，规则为：**只为汉字部分注音，假名部分作为送假名处理，保留在括号外**。
| 单词 | 错误处理 | 正确处理 |
| :--- | :--- | :--- |
| **子供向け** | `子供向け[こどもむけ]` | `子[こ]供[ども]向[む]け` |
| **手続き** | `手続き[てつづき]` | `手[て]続[つづ]き` |
| **取り扱い** | `取り扱い[とりあつかい]` | `取[と]り扱[あつか]い` |

### 3.5 重复词汇独立检查规则 (Repeated Vocabulary Rule)
> **关键规则**：同一词汇在同一文件或多个条目中重复出现时，**每个出现位置都必须独立检查并记录**。

**核心原则：**
- **禁止以"已处理过"为由跳过检查**
- **禁止假设同一词汇的所有出现状态相同**
- **每个 example 都是独立的检查单元**

**常见错误场景：**
```
场景1: 同一词汇在同一条目的多个 example 中重复
条目 N4_0601 (将来):
  example0: 「将[しょう]来[らい]の夢」→ ✅ 已逐字注音
  example1: 「将来[しょうらい]のこと」→ ❌ 整体注音，必须记录
  example2: 「将[しょう]来[らい]性」→ ✅ 已逐字注音
  
错误做法："example0 和 example1 都是'将来'，已经检查过了"
正确做法：example0、example1、example2 各自独立检查，各自独立记录

场景2: 同一词汇在不同条目中重复
条目 N4_0601: 「将[しょう]来[らい]」→ ✅ 已处理
条目 N4_0723: 「将来[しょうらい]」→ ❌ 必须独立检查，不能跳过

错误做法："N4_0601 已经处理过'将来'了"
正确做法：N4_0723 的 example 必须独立检查，状态可能不同
```

**执行要求：**
1. 扫描时必须逐个 example 检查，不能按词汇去重
2. 清单中必须记录每个 example 的完整路径（条目ID + example索引）
3. 审查阶段也必须逐条检查，不能跳过"看起来已经正确的"example

## 5. 内容完整性与非注音修正边界 (Content Integrity & Safety)

### 5.0 执行范围约束 (Scope Limitation)
> **[最高优先级]** 自动化脚本的任何修改操作**必须**严格限制在 `examples` 字段数组的字符串内容中。严禁以任何形式读取、修改、或写入其他任何字段，包括 `expression`, `kana`, `meaning` 等。

### 5.1 严禁操作 (Strict Prohibitions)
AI 系统在执行注音修复时，必须遵守以下"不伤害"原则：
1.  **文本保留 (Text Preservation)**：严禁删除或修改任何语法说明、例句、翻译或接续规则。不得对原文的意思进行"润色"、重写或摘要。
2.  **元数据保留 (Metadata Preservation)**：必须完整保留例句后的真题来源标记（如 `[2010年7月]`、`★` 等），不得视作杂乱字符删除。
3.  **结构完整 (Structural Integrity)**：不得改变 JSON 字段结构或 Markdown 列表层级。

### 5.2 允许修正的明显错误 (Allowed "Obvious Error" Corrections)
仅在以下情况属于**客观且无争议的错误**时，允许顺带修正：

1.  **标点符号规范化 (Punctuation)**
    *   **中英文标点混用**：中文翻译字段中出现的英文逗号/句号及括号，应修正为中文全角标点。
        *   `意思,用法` -> `意思，用法`
        *   `translation.` -> `translation。`
        *   `word(context)` -> `word（context）`
    *   **日文标点**：确保日文例句使用全角 `。` 和 `、`。

2.  **明显假名拼写错误 (Kana Typos)**
    *   **小写假名误用**：修正非拗音/促音位置的错误小写。
        *   `あおぃ` (wrong) -> `あおい` (correct)
    *   **长音符误用**：修正用连字符代替长音符的情况（视情况）。

3.  **格式与配对错误 (Formatting)**
    *   **括号不闭合**：补全遗漏的右括号 `）` 或 `]`。
    *   **残留标签**：删除明显的 HTML 残留代码（如 `<br>` 在纯文本字段中）。
    *   **多余空格**：删除单词内部无意义的空格（如 `日 本 語` -> `日本語`）。

4.  **字符宽度 (Full/Half-width)**
    *   **字母/数字**：统一修正为半角字符（ASCII），以便于搜索系统索引（除非是特意保留的展示用全角）。

5.  **历史性假名用法修正 (Historical Kana)**
    *   **规则**：将旧式或错误的 `ぢ` 和 `づ` 修正为现代标准的 `じ` 和 `ず`，除非它们出现在连浊（Rendaku）的特定情况中（如 `鼻血[はなぢ]`、`三日月[みかづき]`）。
    *   **示例**: `地面[ぢめん]` -> `地面[じめん]`

6.  **标点符号前空格清理 (Whitespace Before Punctuation)**
    *   **规则**: 移除日文或中文标点符号（`。`、`、`、`，`、`！`、`？`）之前的多余空格。
    *   **示例**: `これはペンです 。` -> `これはペンです。`

## 6. AI 实施检查清单 (Implementation Checklist)

1.  **Scanning (扫描)**：识别 `examples` 字段中的汉字序列及现有注音。
2.  **Validation (验证)**：
    *   检查是否包含非汉字字符（如送假名）在括号内。
    *   检查是否为"数字+汉字"或"外来语+汉字"结构。
3.  **Execution (执行)**：
    *   **Step 1**: 匹配 **熟字训白名单**。命中 -> `整体注音`。
    *   **Step 2**: 匹配 **准熟字训灰名单**。命中 -> `整体注音` (可选)。
    *   **Step 3**: **送假名剥离**。将词尾或词中的平假名移出括号。
    *   **Step 4**: **逐字拆分**。应用词典匹配和多音字优先级逻辑。
4.  **Confidence Scoring (置信度标记)**：
    *   🔴 **Low (低)**：未能找到明确拆分/拆分后剩余假名/多音字极度模糊 -> **保持/回退到整体注音**，标记需人工复核。
    *   🟡 **Medium (中)**：规则拆分成功，但属于生僻组合。
    *   🟢 **High (高)**：完全匹配词典库或标准音读规则。
5.  **多轮审查 (Multi-Round Review)** - 详见第8章
    *   **[强制步骤]** 修复完成后必须进行多轮手动审查
    *   每轮审查必须明确标注轮次，逐条手动检查
    *   连续两轮无问题才算完成

### 6.2 扫描阶段强制检查清单 (Scanning Checklist)
> 本清单用于确保扫描阶段不遗漏任何需要修改的 example。

**每个条目的强制检查流程：**
```
条目 N4_XXXX:
  □ example0: 检查所有汉字注音
  □ example1: 检查所有汉字注音
  □ example2: 检查所有汉字注音（如存在）
```

**检查项清单：**
- [ ] **复合词整体注音**：连续2个及以上汉字是否整体注音（如 `銀行[ぎんこう]`）
- [ ] **动词送假名**：动词/形容词词尾假名是否在括号内（如 `歩[あるく]`）
- [ ] **叠字符号**：`々` 是否缺少注音
- [ ] **片假名误注音**：片假名是否被错误添加注音
- [ ] **外来语修饰汉字**：修饰外来语的汉字是否注音
- [ ] **括号配对**：所有 `[` 是否有对应的 `]`
- [ ] **空格违规**：汉字与 `[` 之间是否有空格

**禁止行为：**
1. **禁止抽样检查**：必须检查每个条目的**所有** example，不能抽查
2. **禁止跳过重复词汇**：同一词汇在同一文件中出现多次时，**每次出现都必须独立检查并记录**
3. **禁止依赖记忆**：不能因为"这个词汇刚才检查过"而跳过检查

**重复词汇处理示例：**
```
条目 N4_0601 (将来):
  example0: 「将[しょう]来[らい]」→ ✅ 已逐字注音
  example1: 「将来[しょうらい]」→ ❌ 需要拆分，必须记录
  example2: 「将[しょう]来[らい]性」→ ✅ 已逐字注音

错误做法：example1 因为"和 example0 是同一个词"而跳过
正确做法：example1 必须独立检查并记录到清单
```

## 7. 手动修改操作守则 (Manual Editing Protocol)
> 本守则适用于 AI 在对话中逐条修改 `examples` 字符串时的**强制流程**。

### 7.1 四阶段强制流程
```
[阶段1: 分析] → [阶段2: 列清单] → [阶段3: 锁定并逐条修改] → [阶段4: 双向复核]
```

**阶段1：分析 (Analysis)**
- AI 只能读取 `examples` 字段，扫描所有注音问题
- 输出：问题清单草案（不含修改方案，只列问题类型和位置）

**阶段2：列清单 (Planning)**
- 基于阶段1，生成「修改计划表」(Modification Plan)
- 每条必须有：唯一ID、文件路径、条目索引、example索引、before原文、after修改后、规则依据、置信度
- **清单即合同**：用户确认后，**只允许修改清单中列出的条目**

**阶段3：锁定并逐条修改 (Locked Execution)**
- 用户确认清单后，AI 进入「锁定模式」
- **硬约束**：
  - 只能修改清单中列出的 `before` → `after` 对
  - 修改前必须验证当前文件内容 == `before`，否则停止并报错
  - 一次只改一条，立即 diff 展示
  - 修改后立即用 `Read` 工具确认写入成功

**阶段4：双向复核 (Bidirectional Verification)**
- **正向复核**：清单中的每条是否都已修改？（防遗漏）
- **反向复核**：实际修改的内容是否在清单中？（防误改/多改）
- **结构复核**：JSON 可解析、括号配对、来源标记保留、只改了 `examples`

### 7.2 清单格式规范 (Plan Schema)
每条修改必须包含：
```json
{
  "id": "N2_001_2",           // 唯一ID: 级别_条目号_example序号
  "file": "word_n2_part01.json",
  "item_index": 12,
  "example_index": 2,
  "before": "銀行[ぎんこう]へ行く。",
  "after": "銀[ぎん]行[こう]へ行く。",
  "rule": "Monoruby split",
  "confidence": "high",       // high/medium/low
  "status": "pending"         // pending/done/failed/skipped
}
```

### 7.3 禁止行为 (Prohibited Actions)
1. **清单外修改**：未在计划表中的条目，即使发现新问题，也必须先追加到清单并获用户确认后才能修改
2. **批量落盘**：一次 `Write` 或 `SearchReplace` 只能修改一条 example
3. **跳过验证**：修改后必须用 `Read` 工具二次确认，不能假设写入成功
4. **静默修改**：所有修改必须当场展示 diff，不能攒到最后一起展示

### 7.4 回滚机制 (Rollback)
- 用户可随时说「撤销 ID:xxx」或「撤销上一条」
- AI 必须立即将该条 `after` 还原为 `before`
- 用户说「全部回滚」时，必须还原本次会话所有修改

## 8. 多轮审查机制 (Multi-Round Review Mechanism)
> 本章规定修复完成后必须进行多轮手动审查，确保无遗漏。

### 8.0 核心原则
> **严禁使用任何脚本进行批量审查或自动验证。**  
> 所有审查必须由 AI **逐条手动检查**，每轮审查必须明确标注**第几轮**，并展示详细结果。  
> 只有当前轮次审查发现的所有问题都修复完毕后，才能开始下一轮审查。  
> 以此类推，直到连续两轮审查都没有发现问题为止。

### 8.1 审查轮次流程

```
【第一轮审查】→ 修复所有问题 → 【第二轮审查】→ 修复所有问题 → 【第三轮审查】→ ... → 无问题 → 完成
```

**每轮审查的强制要求：**
1. **明确标注轮次**：必须在输出中清晰标注「第X轮审查」
2. **逐条手动检查**：使用 Read 工具逐个检查 entries，禁止使用脚本批量扫描
3. **记录所有问题**：发现的所有问题必须列出，包括条目ID、example索引、问题描述
4. **修复后才能下一轮**：当前轮次发现的所有问题必须全部修复，才能开始下一轮
5. **连续两轮无问题才算完成**：必须连续两轮审查都没有发现问题，任务才算完成

### 8.1.1 审查阶段发现清单外问题的处理
> **严格禁止顺手修复！**

如果在审查阶段发现**清单外的问题**（即不在当前修改计划表中的问题）：

**必须执行的流程：**
1. **立即停止当前审查**
2. **列出所有新发现的问题**，格式如下：
   ```
   【审查阶段发现清单外问题】
   ====================================
   条目 N4_0603:
     - example2: 食卓[しょくたく] → 应拆分为 食[しょく]卓[たく]
   
   条目 N4_0604:
     - example0: 植物[しょくぶつ] → 应拆分为 植[しょく]物[ぶつ]
     - example1: 植物[しょくぶつ] → 应拆分为 植[しょく]物[ぶつ]
   
   总计: 3处清单外问题
   ```
3. **等待用户确认**：询问用户是否将这些新问题追加到清单
4. **用户确认后**：
   - 将新问题追加到「待追加清单」
   - **重新开始新一轮审查轮次**（从第一轮重新开始）
   - 不能继续当前轮次

**禁止行为：**
- ❌ 禁止顺手修复清单外问题
- ❌ 禁止"既然发现了就一起修了吧"的心态
- ❌ 禁止不告知用户就默默修复
- ❌ 禁止将清单外问题推迟到"下次再处理"

**为什么必须这样？**
- 清单是"修改合同"，修改合同外内容属于"违约"
- 审查阶段的目标是"验证清单执行是否到位"，不是"发现新问题"
- 顺手修复会破坏流程的可追溯性和可验证性

### 8.2 每轮审查的具体内容

#### 第一轮审查：复合词拆分检查
**目标**：检查所有复合词是否已按逐字注音（Monoruby）格式拆分

**检查方法**：
- 手动读取每个 entry 的 examples
- 识别所有连续2个或以上汉字且整体注音的情况
- 对照灰名单判断是否需要拆分

**输出格式**：
```
【第一轮审查】复合词拆分检查
====================================
条目 N4_0123:
  - example0: 銀行[ぎんこう] → 应拆分为 銀[ぎん]行[こう]
  - example1: 合格[ごうかく] → 应拆分为 合[ごう]格[かく]

条目 N4_0124:
  - example0: 問題なし ✅

发现问题: 2处
状态: 需要修复
```

#### 第二轮审查：括号配对和格式检查
**目标**：检查所有括号是否正确配对，格式是否规范

**检查方法**：
- 检查每个注音的 `[` 和 `]` 是否配对
- 检查汉字与 `[` 之间是否有空格
- 检查片假名是否被错误注音
- 检查平假名是否被错误注音

**输出格式**：
```
【第二轮审查】括号配对和格式检查
====================================
条目 N4_0123:
  - example0: 括号配对 ✅
  - example1: 发现空格违规「銀行 [ぎんこう]」→ 应为「銀行[ぎんこう]」

发现问题: 1处
状态: 需要修复
```

#### 第三轮审查：送假名剥离检查
**目标**：检查动词/形容词的送假名是否正确剥离到括号外

**检查方法**：
- 识别动词/形容词词尾的送假名（く、す、る、い、う等）
- 确认送假名在括号外，而非括号内

**输出格式**：
```
【第三轮审查】送假名剥离检查
====================================
条目 N4_0123:
  - example0: 歩[ある]く ✅ 送假名在括号外
  - example1: 歩[あるく] → 应为 歩[ある]く

发现问题: 1处
状态: 需要修复
```

#### 第四轮及以后：综合复查
**目标**：全面复查，确保前序轮次没有遗漏

**检查方法**：
- 重复前三轮的所有检查项
- 特别关注之前修复过的条目
- 检查是否有新引入的问题

### 8.3 审查完成标准

**任务完成必须同时满足**：
1. ✅ 连续两轮审查都没有发现问题
2. ✅ 所有已发现问题都已修复
3. ✅ JSON 结构验证通过
4. ✅ 所有必需字段完整

**最终输出格式**：
```
🎉 多轮审查完成报告
====================================
审查范围: N4_0100-N4_0125
总审查轮次: 4轮

【各轮次结果】
第一轮: 复合词拆分检查 → 发现 3 处问题 ✅ 已修复
第二轮: 括号配对检查 → 发现 1 处问题 ✅ 已修复
第三轮: 送假名剥离检查 → 发现 0 处问题 ✅ 通过
第四轮: 综合复查 → 发现 0 处问题 ✅ 通过

【最终验证】
✅ JSON 结构有效
✅ 所有必需字段完整
✅ 连续两轮无问题

结论: 任务完成，数据质量合格
```

### 8.4 禁止行为

1. **禁止脚本审查**：严禁使用 Python、Shell 等脚本进行批量审查
2. **禁止跳过轮次**：必须按顺序完成每轮审查，不能跳过
3. **禁止虚假报告**：必须真实记录每轮发现的问题数量
4. **禁止提前结束**：必须连续两轮无问题才能结束，不能只进行一轮

## 9. 清单锁定与复核机制 (Plan Lock & Verification)
> 本章是防止「遗漏」和「误删」的核心机制。

### 9.1 清单锁定原则 (Plan Lock Principle)
**清单一旦经用户确认，即成为「修改合同」**：
- ✅ 只能修改清单中列出的条目
- ✅ 只能按清单中的 `before` → `after` 进行修改
- ❌ 禁止修改清单外的任何内容
- ❌ 禁止将 `after` 改为与清单不符的内容

**发现新问题时的处理**：
1. 暂停当前修改
2. 将新问题追加到「待追加清单」
3. 等待用户确认追加后才能修改
4. **严禁顺手修复**

### 9.2 双向复核流程 (Bidirectional Verification)

#### 复核时机
- **每改完5条**：立即执行一次快速复核
- **全部修改完成后**：必须执行最终复核
- **用户要求时**：随时执行复核

#### 复核步骤

**Step 1: 正向复核 (遗漏检测)**
```
检查清单中所有 status=pending 的条目：
- 该条是否已在文件中被修改？
- 如果是 → 标记为 done
- 如果否 → 保留 pending，提示用户「以下条目尚未修改」
```

**Step 2: 反向复核 (误改/多改检测)**
```
扫描文件中所有 examples：
- 找出所有与原始状态不同的 example
- 检查这些修改是否都在清单中有对应条目
- 如果发现「清单外的修改」→ 立即报警，提示用户「发现未授权修改」
```

**Step 3: 结构复核 (误删检测)**
```
对修改过的文件执行：
- JSON 可解析性检查
- examples 数组长度检查（与原始长度对比）
- 每条 example 字符串长度检查（突变 >20% 报警）
- 来源标记正则检查 /\[\d{4}年|\★/ 是否存在
- 括号配对检查 [ ] （）
- 非 examples 字段一致性检查（必须与原文件完全一致）
```

### 9.3 复核报告格式 (Verification Report)
```
=== 复核报告 ===
【正向复核 - 遗漏检测】
✅ 已完成: 15条
⏳ 待修改: 3条 (ID: N2_008_1, N2_015_0, N2_015_2)

【反向复核 - 误改检测】
✅ 未发现未授权修改

【结构复核 - 误删检测】
✅ JSON 可解析: 5/5 文件
✅ examples 长度一致: 5/5 文件
✅ 来源标记保留: 100%
✅ 括号配对: 100%
⚠️ 长度突变警告: word_n2_part03.json item_22 example_1 (长度变化 -35%)

【结论】
🟡 部分通过 - 请检查长度突变警告
```

### 9.4 异常处理 (Exception Handling)

| 异常情况 | 处理方式 |
|---------|---------|
| 正向复核发现遗漏 | 列出未修改条目，询问用户「继续修改/跳过/标记为跳过」 |
| 反向复核发现未授权修改 | **立即停止**，展示未授权修改内容，询问「回滚/接受/追加到清单」 |
| 结构复核发现 JSON 破坏 | **立即停止**，展示错误详情，必须回滚到修改前状态 |
| 来源标记丢失 | **立即停止**，该条必须回滚，重新按规则修改 |
| 括号不配对 | 标记为 failed，需要人工复核 |

### 9.5 用户确认检查点 (User Checkpoints)
以下节点必须获得用户明确确认（不能默认继续）：
1. **清单确认**：「以上修改计划是否确认？确认后进入锁定模式。」
2. **每5条复核**：「已完成5条修改，复核通过，是否继续？」
3. **最终复核**：「全部修改完成，最终复核报告如上，是否确认提交？」
4. **发现异常**：任何异常都必须等待用户决策，不能自动继续

> ⚠️ **核心原则**：宁可多确认一次，不能漏确认一次。用户说「继续」才算继续。
